커리큘럼은 단순히 화면을 그리는 법을 배우는 과정이 아니라,

**문제를 발견하고, 사용자를 이해하고, 구조와 로직과 상태를 설계하며, 실전 협업과 검증, 개선까지 경험하는 과정**으로 설계한다.

전체 흐름

- **초반부**: 기획의 기본기와 설계의 뼈대 형성
- **중반부**: 개인 포트폴리오와 아이디어톤 실전 적용
- **후반부**: 데이터, 비즈니스, AI 실무 활용 확장
- **실전 회고부**: 해커톤 이후 회고와 문제 재정의

## 1주차. UXUI 트랙 이해, 서비스 제작 과정, PM 입문, 벤치마킹과 역기획

### 학습 목표

UXUI 트랙의 전체 방향을 이해하고, 서비스 제작 과정 속에서 기획이 어떤 역할을 하는지 파악한다.

또한 벤치마킹과 역기획을 통해 좋은 서비스를 읽는 기본 관점을 익힌다.

### 학습 내용

- UXUI 트랙 전체 목표
- UX란 무엇인가, UI란 무엇인가
- UX와 UI의 차이
- 웹/앱 서비스 제작 과정 전체 흐름 이해
- FE / BE / PM / Design의 역할 관계
- PM이란 무엇인가
- PM이 서비스 제작 과정에서 맡는 일
- 벤치마킹이란 무엇인가
    - 예쁜 화면 모으기와 벤치마킹의 차이
    - 무엇을 어떤 기준으로 비교해야 하는가
    - 구조, 흐름, 메시지, 예외 처리, 상태 처리 관점에서 보기
- 역기획이란 무엇인가
- 기존 서비스를 읽는 기본 질문
- 간단한 역기획 실습

### 이 주차의 핵심

- 기획자는 서비스를 “느낌”으로 보는 사람이 아니라, **구조와 의도를 읽는 사람**이다.
- 벤치마킹은 복사가 아니라 **비교를 통한 학습과 해석**이다.

## 2주차. Why, 사용자 조사, Who, 페르소나

### 학습 목표

어떤 문제를 누구를 위해 해결할 것인지 정의하는 방법을 익힌다.

추상적인 아이디어를 구체적인 사용자 문제로 바꾸는 사고를 훈련한다.

### 학습 내용

- Why - Who - What - How 프레임워크 간단 소개
- Why: 어떤 Pain Point를 해결할 것인가
- 문제 정의가 중요한 이유
- 사용자 조사 기초
    - 간단한 인터뷰
    - 리뷰/VOC 분석
    - 데스크 리서치
    - 기존 서비스 사용 경험 관찰
- Who: 누가 가장 불편한가
- 타겟을 좁히는 기준
    - 샥즈 예시로 들기!
- 페르소나(Persona)란 무엇인가
    - 넓은 타겟을 설계 가능한 사용자로 구체화하는 도구
    - 문제 해결 중심 페르소나 작성법
- 사용자 경험에 영향을 미치는 요인
    - 외부 환경 요인
    - 내부 갈등 요인
- 사용자 정의와 페르소나 초안 작성 실습

### 이 주차의 핵심

- 모든 문제는 모든 사람의 문제가 아니다.
- 좋은 기획은 **구체적인 사용자와 구체적인 불편을 선명하게 잡는 것**에서 시작한다.

## 3주차. What, MVP, 우선순위 결정

### 학습 목표

정의한 문제와 타겟을 바탕으로 무엇을 만들지 결정하고, 핵심 기능만 남기는 방법을 익힌다.

### 학습 내용

- What: 무엇을 만들 것인가
- How: 어떤 방식으로 접근할 것인가
- MVP란 무엇인가
- MVP를 잘못 이해했을 때 생기는 문제
- 우선순위 결정 기초
    - MoSCoW
- 기능 덜어내기 실습
- “이번에 만들 것 / 이번에는 만들지 않을 것” 정리

### 이 주차의 핵심

- 기획은 아이디어를 늘리는 작업이 아니라, **핵심만 남기는 작업**이다.
- MVP는 기능이 적은 서비스가 아니라, **핵심 문제를 먼저 푸는 최소 구성**이다.

## 4주차. IA, User Flow, 산출물 차이 이해, FigJam 실습

### 학습 목표

서비스의 구조와 사용자 이동 흐름을 설계하는 방법을 이해하고, 여러 산출물의 차이를 정확히 구분할 수 있도록 한다.

### 학습 내용

- IA(Information Architecture, 정보구조)란 무엇인가
- User Flow란 무엇인가
- 구조 설계와 흐름 설계의 차이
- 핵심 성공 경로(Happy Path) 중심 구조 설계
- IA / User Flow / Flow Chart / Story Board / Wireframe 차이 정리
    - IA = 구조
    - User Flow = 사용자 이동
    - Flow Chart = 조건 분기와 로직 표현
    - Story Board = 사용 맥락과 시나리오
    - Wireframe = 화면 초안
- Happy Path를 기준으로 전체 서비스 흐름 보기
- FigJam을 활용한 IA + User Flow 제작 실습

### 이 주차의 핵심

- 이 주차는 **유저가 어디로 가는지** 설계하는 주차다.
- 비슷해 보이는 산출물도 역할이 다르므로 혼용하면 안 된다.

## 5주차. 화면 설계서와 로직 설계 - 시스템 중심 설명

### 학습 목표

구조 설계를 바탕으로 실제 화면 안에서 어떤 일이 일어나는지, 어떤 조건에서 성공과 실패가 갈리는지 설계하는 방법을 익힌다.

### 학습 내용

- 화면 설계서(SB)란 무엇인가
- 화면 설계서에 로직이 필요한 이유
- 데이터 요청 시점 이해
    - on load
    - on interaction
- Event - Condition - Action 구조
- Happy Path / Unhappy Path
- 로직 설계와 예외 처리
- 핵심 예외 설계
    - 입력값 공백
    - 네트워크 끊김
    - 결과 0건
- 추가 예외 설계
    - 유효성 실패
    - 권한 없음
    - 로딩 지연
- Flow Chart는 별도 산출물 본체가 아니라, 로직 분기를 설명하는 보조 표현 도구로 이해

### 이 주차의 핵심

- 이 주차는 **무슨 일이 일어나는지** 설계하는 주차다.
- 실제 서비스는 성공 경로만으로 완성되지 않는다. **실패와 예외를 함께 설계해야 한다.**
